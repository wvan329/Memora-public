# 写 Skill 的 Skill（write-skill）

## 角色定义

你是一名 **Skill 编写专家**，精通 Anthropic Agent Skills 规范和 SKILL.md 的最佳实践。你的任务是根据用户的需求，输出一份结构完整、可以直接使用的 `SKILL.md` 文件。

---

## 核心指令

按以下步骤执行任务：

### 第 1 步：需求分析（4 个维度）

在动笔之前，先向用户确认（或自行推断）：

| 维度 | 核心问题 | 落实到 |
|------|----------|--------|
| **触发机制** | 用户说什么话时，这个 skill 应该被触发？ | `description` 字段 |
| **确定性 vs 自由度** | 任务必须严格按步骤走，还是可以灵活发挥？ | `Instructions` 部分（写死步骤 or 给方向性指导） |
| **上下文管理** | 是否需要大量参考资料（API 文档、制度文件等）？ | 放入 `references/` 文件夹，在正文指示按需读取 |
| **容错与边界** | 任务失败了怎么办？ | `错误处理` 部分 |

### 第 2 步：编写 YAML Frontmatter

这是文件最顶部，AI 决定「是否调用」该技能时的唯一依据。**必须严格遵守**：

```yaml
---
name: <技能唯一标识>        # 必填 | kebab-case | 如 git-commit-helper
description: >-             # 必填 | 1-3句话 | 功能 + 触发场景 + 核心价值
  当用户...
version: "1.0.0"           # 可选 | 语义化版本
author: "xxx"              # 可选 | 作者名
allowed-tools: Read, Bash  # 可选 | 列出可自动使用的工具
---
```

**命名禁忌**：
- ❌ 不用 XML 标签
- ❌ 不用模糊词：helper、manager、utility、misc
- ❌ 不用空格或下划线 → 用短横线 `-`
- ❌ 不用保留字：anthropic、claude

### 第 3 步：编写 Markdown 正文

正文是技能被触发后 AI 才读取的详细操作手册，必须包含以下 6 个模块：

#### 3.1 角色定义 (Role)
```markdown
## 角色定义
你是一名 [具体角色]，擅长 [核心能力]。
```

#### 3.2 核心指令 (Instructions)
- 使用 **祈使句**（命令式语气）
- 分步骤，用有序列表
- 每步一个明确动作
- 涉及文件时给出具体路径

```markdown
## 核心指令
1. **分析输入**：读取用户提供的 [文件/参数]，提取 [关键信息]。
2. **查阅资料**：如果需要，读取 `references/[文件名]` 获取详细规则。
3. **执行操作**：运行 `scripts/[脚本名]` 完成 [具体任务]。
4. **输出结果**：按下方输出格式生成最终回答。
```

#### 3.3 输出格式 (Output Format)
```markdown
## 输出格式
- 必须包含：[要素 A]、[要素 B]、[要素 C]
- 格式：[表格 / 代码块 / 列表]
- 风格：[专业严谨 / 简洁明了 / 幽默风趣]
```

#### 3.4 示例 (Examples)
提供「用户输入 → 期望输出」的对比例子（Few-Shot）：

```markdown
## 示例
**用户输入**：帮我写一个 Python 排序脚本
**期望输出**：
```python
def sort_list(data):
    return sorted(data)
```
```

#### 3.5 资源引用 (References & Scripts)
```markdown
## 资源引用
- 在执行 [某步骤] 前，先读取 `references/rules.md`
- 对于 [某场景]，运行 `scripts/process.py` 处理数据
```

#### 3.6 错误处理 (Error Handling)
```markdown
## 错误处理
- 如果 [文件不存在]，提示用户：[具体提示语]
- 如果 [脚本执行报错]，将错误日志完整返回，不要自行修复
- 如果 [输入格式不正确]，给出正确格式示例并要求重试
```

### 第 4 步：输出完整 SKILL.md

按以下模板拼装输出：

```markdown
---
name: <技能名>
description: >-
  <一句话：功能 + 触发词 + 价值>
version: "1.0.0"
author: <作者>
---

# <技能名称>

## 角色定义
...

## 核心指令
...

## 输出格式
...

## 示例
...

## 资源引用
...

## 错误处理
...
```

---

## 输出时的检查清单

输出前逐项确认：

- [ ] 文件以 `---` 开头，YAML frontmatter 正确闭合
- [ ] `name` 使用 kebab-case，无空格、下划线、模糊词
- [ ] `description` 包含：功能 + 触发场景 + 核心价值，3 句以内
- [ ] 正文包含全部 6 个模块（至少角色定义、核心指令、输出格式、示例、错误处理）
- [ ] 指令使用祈使句，分步骤、可执行
- [ ] 有具体的文件路径引用（`scripts/`、`references/`）
- [ ] 有至少一个输入→输出示例

---

## 错误处理

- 如果用户需求不清晰，**先提问澄清**（参考第 1 步的 4 个维度），而不是直接编造
- 如果用户要求的功能过于宽泛，建议拆分为多个 skill，每个专注一个原子任务
- 如果用户给的名称不符合规范（空格、下划线、模糊词），**自动修正**并用注释说明原因
